# Workflow

## 目标
把“用户从 Telegram 发晚饭照片”这件事，标准化成一条可重复执行的流水线：

1. 接收原始图片
2. 归档到 intake 目录
3. 用 YOLO 分图得到单菜候选
4. 为每个候选生成信息卡片草稿
5. 进入人工确认/网页编辑
6. 写回正式菜品库

## 目录约定
- 原始图片：`dinner-what-to-eat/uploads/intake/<YYYY-MM-DD>/`
- 分图结果：`dinner-what-to-eat/uploads/.../yolo-splits/`
- 菜品库：`dinner-what-to-eat/data/dishes.json`
- 候选库：`dinner-what-to-eat/data/dish-split-candidates.json`

## 标准处理顺序

### 1. 获取图片
如果用户在 Telegram 里上传新图片：
- 先把图片保存到 workspace 的 intake 目录
- 保留原始文件名和来源时间
- 不直接覆盖历史 JPG/候选文件

### 2. 运行分图
优先复用已有脚本：
- `dinner-what-to-eat/scripts/yolo_plate_segment.py`

如果用户给的是一批新图，而现有脚本写死了输入目录：
- 先复制到脚本约定的输入目录，或
- 先修改脚本使其支持参数化输入输出目录，再运行

### 3. 生成候选信息卡片
每个候选至少要有：
- `id`
- `name`（允许先是待确认）
- `photo`
- `sourcePhoto`
- `category`
- `ingredients`
- `prepTimeMin`
- `caloriesKcal`
- `nutritionNote`
- `needsConfirm`

如果无法高置信识别：
- 允许保留“待确认”占位，但必须把图片和最基础元信息写进去
- 不要凭空编造很具体的食材细节

### 4. 进入网页编辑
网页端要支持两类编辑：
- **菜品层编辑**：修改菜名、材料、营养、时间、图片归档信息
- **套餐层编辑**：一顿饭由多道菜组成，可手动勾选蛋白质/蔬菜/主食

### 5. 写回正式库
确认后的菜品：
- 写入 `dishes.json`
- 清除 `needsConfirm`
- 保留来源图片/裁剪图路径

## 页面设计约束
“晚饭吃什么”页面后续要从“单菜随机推荐”升级为“套餐推荐”。

### 套餐结构
一顿晚饭应至少支持：
- 蛋白质（必选或优先）
- 蔬菜
- 主食

### 自动检查清单
用户选好套餐后，自动生成材料检查清单：
- 可按分类分组（蛋白质 / 蔬菜 / 主食）
- 默认自动勾选推荐材料
- 允许用户手动取消/补充
- 允许后续扩展成“缺什么就补什么”的购物清单

## 风险控制
- 不要覆盖用户原始照片
- 不要把低置信度图像识别结果直接当成最终事实
- 对“待确认菜名”保留人工确认入口
- 如果脚本是批量写入，先输出候选，再合并到正式库，避免一次性污染 `dishes.json`
